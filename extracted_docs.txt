RESQLINK
Disaster Relief Coordination Platform
Product Requirements Document  |  v1.0


1. Project Overview
ResQLink is a web-based disaster relief coordination platform designed to bring structure, efficiency, and real-time intelligence to post-disaster relief operations. When a disaster strikes and survivors move to safe zones, the current model of help is chaotic — people show up without direction, resources are duplicated or wasted, and urgent needs go unmet.

ResQLink solves this by giving every relief actor — doctors, pharmacies, volunteers, and donors — a dedicated, role-specific interface powered by AI task assignment. The platform ties all of these roles together in a unified coordination layer.


2. User Roles & Personas
The platform has 5 primary user roles, each with a distinct registration flow, dashboard, and feature set.



3. Site Map & Navigation
All pages are accessible from a single-page application (SPA) structure with role-based routing. Unauthenticated users land on the Home page. After login, the user is redirected to their role-specific dashboard.


4. Detailed Feature Specifications

4.1.2 Role Selector

Default selected: Volunteer

4.1.3 Login Forms
Each role has a distinct set of registration/login fields. On first visit the form acts as registration; return users authenticate with phone + OTP or password.

4.3 Doctor — Site Page  /site/:id/doctor
When a doctor clicks 'Volunteer Here' on a site card, they go to the Doctor site page.

4.3.1 Patient Tags
Each patient at the site is represented as a collapsible tag card (Patient ID + triage level)
Doctor can expand any patient tag and fill in:
Medicines needed (name, dosage, quantity)
Medical equipment needed (IV, oxygen, etc.)
Priority level (Urgent / Moderate / Stable)
Notes / instructions for nurse
Submission sends the request to the pharmacy queue
Doctors can see which requests are fulfilled vs. pending

4.4 Pharmacy — Site Page  /site/:id/pharmacy
After clicking 'Volunteer Here', pharmacy goes to their site detail page.

4.4.1 Order Queue
Pharmacy selects which site they are supplying (dropdown if multiple)
Sees list of pending orders from doctors at that site
Each order shows: Patient ID, medicine/equipment name, quantity, priority
Pharmacy can mark individual items as: Available / Partially Available / Unavailable
On confirmation, order is flagged for driver pickup

4.4.2 Driver Dispatch
Once an order is packed, status changes to 'Ready for Pickup'
AI assigns the nearest available driver automatically
Pharmacy sees driver name + ETA on the order card

10. Key UX & Design Principles
Speed is survival: All critical actions must be reachable in 3 taps or fewer
Mobile-first: Design for small screens; most users will be on phones in the field
High contrast: Use large fonts (min 16px body), strong color contrast for outdoor sunlight readability
Offline-awareness: Show clear 'You are offline' banners; cache last known site data
No jargon: All labels in plain, simple language. Avoid medical/technical terminology in UI
Emergency numbers always visible: Chatbot button always in header, never hidden
AI is assistive, not blocking: AI task assignments can be overridden by site admins or senior volunteers
Urgency is visual: Red/orange/green color coding must be consistent across all pages

This clarifies the entire architecture perfectly. With exactly three volunteer tracks, a placeholder for the payment gateway, and a clear understanding that patient tags act purely as offline identifiers for logistics, the system's logic is locked in.
To make this platform highly organized for people moving from danger zones to safe zones, the underlying web development stack will need a robust backend. Using Python alongside advanced data structures—particularly graph algorithms for the delivery routing, similar to the networking logic used in a "Six Degrees of Separation" model—will be ideal for handling the AI dispatch matching.

Here is the complete, exhaustive blueprint of your disaster management web application from front to back.

🌐 Global Interface & Navigation
The application features a persistent home page header displaying your custom logo in the top left corner.

The top right corner prominently features a "Donate" option.

Directly adjacent to the donate button is a "Help Me" feature designed for immediate crisis intervention.

Clicking "Help Me" triggers a chatbot that immediately provides users with emergency phone numbers.

The chatbot outputs real-time data regarding the nearest safe spots and provides instructions on immediate safety measures to take.

🔐 Authentication & Role-Based Onboarding
The main landing page requires users to select one of three specific pathways: Doctor, Pharmacy, or Volunteer.

Volunteer Onboarding (Default View):
The form captures name, phone number, profession, date of birth, and blood group.

Volunteers specify their vehicle availability: whether they are an active driver, or if they are simply donating the use of an empty vehicle.

Volunteers list any medical equipment they already possess and verify if they are medically fit.

Volunteers state their on-site availability duration and any prior knowledge in disaster management.

Based on their input, they can list immediate item demands that a driver can transport to the site.

Doctor Onboarding:
Selecting "Doctor" shifts the form to capture name, phone number, date of birth, blood group, and medical specialty.

Pharmacy Onboarding:
Selecting "Pharmacy" shifts the form to capture name, phone number, the physical address of the pharmacy, date of birth, and blood group.

🎛️ The Command Center (Main Dashboard)
Following login, all users are routed to a centralized home dashboard displaying active disaster sites.

Sites are formatted as UI cards showing the site name and the current number of patients.

Each card lists urgent personnel requirements, highlighting specific deficits in doctors, nurses, or drivers.

The bottom of every site card contains a CTA (Call to Action) button labeled "Volunteer".

🩺 1. The Medical Workflows (Doctor & Pharmacy)
When medical personnel click the "Volunteer" CTA on a site card, they enter highly specialized operational workspaces.
The Doctor Workflow:
Doctors are directed to a specialized interface displaying offline-assigned patient tags for their chosen site.

Because victims cannot operate the site, these tags serve as the primary routing keys for medication delivery.
For each patient tag, the doctor digitally writes out specific needs, requesting medicines and equipment from the pharmacy.

The Pharmacy Workflow:
Pharmacies enter a dedicated page where they choose the specific site they are supplying.

They view a real-time ledger of the specific medical orders submitted by the doctors.

Pharmacies pack the orders accordingly, queuing them in the system to be picked up by a driver volunteer.

⚙️ 2. The Volunteer Engine (AI Dispatch)
When a general user clicks the "Volunteer" CTA on a site card, an AI engine analyzes their provided profession data and assigns them one of three specific operational tracks.

Track 1: The Nurse Interface
Nurses are routed to a distinct dashboard where they can submit demands for supplies.

Nurse demands are restricted to basic first aid items; they cannot request heavy medicines.

They view the exact quantity of resources currently available on site.

They utilize a digital to-do list to associate patient tags with the appropriate physical measures they need to administer.

They have an integrated call option to contact and coordinate with tending volunteers assigned under them.

Track 2: The Vehicle & Logistics Interface
Vehicle operators are given a separate logistics hub.

Active Drivers: The AI generates a choice list of pharmacies and sites to ensure the equal distribution of donated resources.

Urgent Overrides: If a site faces a critical, urgent requirement, the AI overrides the choices and issues a mandatory, single-site delivery route to the driver.

Priority Sorting: The driver's task list is color-coded and sorted in descending order of emergency (red for most urgent, green for least urgent).

Navigation: Upon selecting a site, the driver receives precise pickup/drop-off details mapped onto an in-app interface for ease.

Vehicle Only: If a user only has a vehicle, they input the vehicle type and current address; the AI then assigns an available driver to it.

Skill Only: If a user can drive but has no vehicle, the AI automatically pairs them with an idle vehicle.

Track 3: The Helping Volunteer Interface
General helpers are routed to a separate coordination page.

The AI evaluates their data and assigns them a specific operational lead (e.g., a nurse, head chef, or tent-making team).

Helpers are given the exact geographic location to report to.

Helpers are issued a suggested to-do list of tasks.

Helpers have tools to raise complaints regarding their workspace, teammates, or leads.

Helpers can submit operational suggestions.

Helpers utilize a call function to connect directly with their assigned lead.

📦 3. The Donation & Resource Routing Engine
Activating the "Donate" button opens a portal with two distinct tracks: monetary or physical goods.

Monetary Donations: This tab will feature an open interface prepared to integrate an external payment gateway for processing financial contributions.
Physical Goods ("Others"): This track routes users to an inventory input page.

Users enter the specific type of donation (e.g., clothes, food), which the AI automatically categorizes.

Users enter the exact quantity.

Users choose if they want to deliver the items themselves.

If self-delivering, the system provides the safe zone location and offers the user the option to register as a driver volunteer, routing them into the logistics workflow.

If they cannot deliver, the system flags the items and automatically dispatches a driver volunteer to their location for pickup.
